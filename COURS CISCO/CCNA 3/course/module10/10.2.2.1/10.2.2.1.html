Résoudre les problèmes de voisinage

Cet exemple illustre comment résoudre des problèmes de voisinage. Dans la topologie de la figure 1, tous les routeurs ont été configurés pour prendre le routage OSPFv2 en charge.

En observant rapidement la table de routage de R1, comme illustré à la Figure 2, vous remarquez qu'aucune route OSPFv2 n'est ajoutée. Cela peut être causé par de multiples raisons. Cependant, une condition requise pour l'établissement d'une relation de voisinage entre deux routeurs est la connectivité de couche 3 OSI.

La Figure 3 confirme que l'interface S0/0/0 est activée et qu'elle fonctionne. La réussite de la requête ping confirme également que l'interface série de R2 est active. Une requête ping qui aboutit ne signifie pas qu'une contiguïté s'établira, parce qu'elle a peut-être chevauché des sous-réseaux ou mal configuré des paramètres OSPF. Vous devez encore vérifier que les interfaces sur les périphériques connectés partagent le même sous-réseau. En cas d'échec de la requête ping, vérifiez le câblage et vérifiez que les interfaces sur les périphériques connectés sont configurées correctement et qu'elles fonctionnent.

Pour qu'une interface soit activée pour OSPFv2, une commande network correspondante doit être configurée sous le processus de routage OSPFv2. Vous pouvez vérifier les interfaces OSPFv2 actives avec la commande show ip ospf interface. Le résultat présenté dans la Figure 4 confirme que l'interface série 0/0/0 est activée pour OSPFv2. Si les interfaces connectées de deux routeurs ne sont pas activées pour OSPF, les voisins ne peuvent pas établir de contiguïté.

Vérifiez les paramètres OSPFv2 avec la commande show ip protocols. Le résultat présenté dans la Figure 5 confirme que le protocole OSPFv2 est activé et dresse la liste des réseaux annoncés comme étant activés par la commande network. Si une adresse IPv4 sur une interface fait partie d'un réseau qui a été activé pour OSPFv2, l'interface sera activée pour OSPFv2.

Cependant, notez que l'interface Serial 0/0/0 est répertoriée comme passive. N'oubliez pas que la commande passive-interface stoppe les mises à jour de routage entrantes et sortantes, car elle entraîne l'arrêt par le routeur de l'envoi et de la réception des paquets Hello sur l'interface concernée. Pour cette raison, les routeurs ne deviennent pas des voisins.

Pour désactiver une interface passive, utilisez la commande de mode de configuration de routeur no passive-interface comme illustré à la Figure 6. Dès que vous avez désactivé l'interface passive, les routeurs deviennent contigus, comme indiqué par le message d'information produit automatiquement.

En analysant rapidement la table de routage, comme illustré dans la Figure 7, vous voyez que le protocole OSPFv2 échange désormais des informations de routage.

Un autre problème peut survenir lorsque deux routeurs voisins présentent des tailles de MTU (Maximum Transmission Unit, unité de transmission maximale) différentes sur leurs interfaces de connexion. La taille MTU est la taille maximale des paquets de couche réseau transférés par le routeur sur chaque interface. La taille MTU par défaut d'un routeur est de 1 500 octets. Cependant, cette valeur peut être modifiée pour les paquets IPv4 via la commande de configuration d'interface ip mtu size ou la commande d'interface ipv6 mtu size pour les paquets IPv6. Si deux routeurs connectés comportent des MTU différentes, ils tentent d'établir une contiguïté, mais ils ne réussissent pas à échanger leurs LSDB (Link-state Database, base de données d'états de liens) et la relation de voisinage échoue.
